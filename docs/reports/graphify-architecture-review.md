# Graphify Architecture Review

**Date:** 2026-06-30  
**Phase:** 2 (ADR-001)

## Current Dependency Graph Analysis

Based on the latest `graph.json` generated by Graphify, EcoFleet Express currently clusters around two massive central nodes:
1. `core.models` (Specifically the `*Workbook` models)
2. `core.operations.providers` (Specifically the Health Check providers)

### Findings:
- The `core.views` module exhibits high cyclic dependency with `core.workbook` because the controllers directly invoke the file lock primitives (`core.workbook.locking.workbook_lock`) and atomic save helpers. 
- The entire `core.attendance`, `core.btpl`, and `core.ftl` modules are heavily bound to `openpyxl`. There is no abstraction layer between the business logic and the Excel parsing logic.
- The `Operations Center` is heavily decoupled from the database; its `Provider` architecture relies almost exclusively on reading OS-level file attributes and executing `len(wb.sheetnames)` to assess system state.

## Future Architecture Implications

The Phase 2 ADR (Operational Data Migration) fundamentally changes this dependency graph.

### Areas that will change in Phase 3:
- **`core.models`**: Will explode in size with the addition of normalized Domain Models (`Shipment`, `Driver`, `Vehicle`).
- **`core.*` (Business Logic)**: The direct imports of `openpyxl` within `core.btpl` and `core.ftl` will be extracted into a new `core.importers` module. The business logic will strictly import Django ORM querysets.
- **`core.views`**: The lock primitives (`workbook_lock`) will be removed from controller flows entirely, massively reducing view complexity.

### Areas intentionally left untouched:
- **`core.operations.providers`**: The `Provider` hierarchy (`BaseProvider`, `FTLProvider`, etc.) will remain exactly the same. The internal logic of `FTLProvider.check_health()` will simply change from an Excel read to a PostgreSQL `COUNT(*)` query. This ensures the Operations Center dashboard UI requires zero modifications.
- **Frontend Assets**: `main.js` and the Django templates rendering the portal are decoupled from the data storage mechanism. They will remain untouched.

## Summary

The current architecture is overly coupled to the filesystem. The ADR decouples this by injecting a database layer between the business logic and the filesystem. Graphify reveals that by enforcing a strict Provider interface, we can swap out the backend engine (Excel to PostgreSQL) without needing to rewrite the top-level Operations Center or Portal UI components.
