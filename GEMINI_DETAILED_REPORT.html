EcoFleet Express

Comprehensive Enterprise Software Analysis Report

Overall Project Rating: 7.5 / 10

Justification: EcoFleet Express exhibits a clean MVC structure built on Django with high internal code quality. The recent migration to PostgreSQL and integration of Celery/Redis for background processing resolves its most severe architectural bottleneck (synchronous Excel locking). However, critical authorization gaps (IDOR), legacy deployment footprints (reliance on runserver, local media storage), and scattered business logic prevent it from being cloud-native and production-ready. With the execution of the final structural cutovers (Sprint 6), this project will be a highly scalable and robust internal tool.

Implemented Features

1. FTL & BTPL Shipment Tracking

Complete shadow-import mechanics into PostgreSQL, supporting dual-read operations, celery-based async uploads, and Excel reconciliation. Currently in the dual-write soak phase.

2. Pincode Engine

Fast lookup engine backed by a 20,000+ row dataset allowing users to identify operational zones for logistics planning.

3. Attendance Management

Tracks driver operational days. The Employee model schema has been prepped for a unified HR integration.

4. Secure Authentication Portal

Django-backed authentication with comprehensive HTTP header enforcement and audit logging (SystemEvent).

5. File Export Generation

Automated Excel sheet synthesis for fleet operations and reporting using openpyxl.

Design Flaws

DES-001: Fragmented Date Parsing Logic

Description: The date parsing logic (converting strings and openpyxl datetimes into standardized datetime.date) is duplicated between excel_importer.py and btpl_importer.py. This violates DRY principles and causes repeated bugs.

Location: core/importers/excel_importer.py (Lines 98-120), core/importers/btpl_importer.py (Lines 74-90)

# Proposed Solution # Move to a shared utility `core/utils/date_parser.py` def parse_excel_date(date_val): if isinstance(date_val, str): try: return dateutil.parser.parse(date_val).date() except Exception: return None elif isinstance(date_val, datetime.datetime): return date_val.date() return date_val

DES-002: In-Memory File Locks (Redundant)

Description: The custom core.workbook.locking module uses a filesystem lock to prevent concurrent Excel writes. With PostgreSQL, this is an anti-pattern. Database-level row locking (select_for_update()) is much safer.

Location: core/workbook/locking.py (Lines 1-50)

# Proposed Solution # Remove `locking.py` entirely in Phase 3. Rely purely on PostgreSQL MVCC and transactional integrity.

Architecture Flaws

ARCH-001: Stateful Single-Node Media Architecture

Description: Workbooks are stored locally in the /media/ directory. If the application scales to 2+ web servers, one server will not have access to the workbooks uploaded on the other server, breaking downloads and rendering horizontal scaling impossible.

Location: ecofleet/settings.py (MEDIA_ROOT, Line 131)

# Proposed Solution # Integrate `django-storages` with AWS S3 or an equivalent object store. # settings.py: DEFAULT_FILE_STORAGE = 'storages.backends.s3boto3.S3Boto3Storage' AWS_STORAGE_BUCKET_NAME = 'ecofleet-media'

ARCH-002: Missing WSGI/ASGI Production Server

Description: The infrastructure depends on Django's built-in runserver. This is strictly forbidden in production due to massive performance, threading, and security limitations.

Location: Deployment scripts / Runtime env

# Proposed Solution # Introduce Gunicorn and a reverse proxy (Nginx). pip install gunicorn gunicorn ecofleet.wsgi:application --workers 4 --bind 0.0.0.0:8000

Security Flaws

SEC-001: IDOR in FTL & COF Workbook Access (Critical)

Description: The endpoints to view, download, or edit FTL and COF workbooks lack authorization boundaries. A logged-in user can modify URL parameters to access another logistics provider's sensitive shipment data.

Location: core/views/ftl.py and core/views/cof.py

# Proposed Solution # Apply a `@provider_required` or strict ownership check in the view. from django.shortcuts import get_object_or_404 def ftl_sheet(request, pk): workbook = get_object_or_404(FtlWorkbook, pk=pk, owner=request.user) # ... proceed with logic

SEC-002: Unsafe Inline Content Security Policy (High)

Description: script-src in the CSP allows 'unsafe-inline'. This exposes the application to severe Cross-Site Scripting (XSS) vulnerabilities if any user input is rendered unescaped on the frontend.

Location: ecofleet/settings.py (Line 101)

# Proposed Solution # Implement CSP nonces or strict hashes for all inline scripts. CSP_SCRIPT_SRC = ("'self'", "'nonce-{request.csp_nonce}'")

SEC-003: Unverified Proxy SSL Header Trust (Medium)

Description: Django trusts the X-Forwarded-Proto header blindly via SECURE_PROXY_SSL_HEADER. If the application is ever exposed directly to the internet without an edge proxy stripping malicious headers, an attacker can spoof HTTPS, bypassing secure cookie flags.

Location: ecofleet/settings.py (Line 95)

# Proposed Solution # Ensure deployment topology enforces a strict reverse proxy layer. # Only set SECURE_PROXY_SSL_HEADER if the proxy is confirmed to sanitize it. # Alternatively, enforce IP-based proxy trust middleware.
